# ADA Order Pilot

面向跨境奶粉业务的智能订单系统原型。核心目标不是“做一个能跑的接口”，而是把**不稳定输入**变成**可控输出**，并在系统层面保证一致性、可追踪性、可扩展性。

---

## 1. 系统定位（System Framing）

### 1.1 业务问题
- 订单来源是非结构化文本（聊天、手工录入、截图转录），信息噪声高。
- 商品名称存在别称、缩写、品牌口语化表达。
- 收件人信息经常不完整，但历史库中可能已有可复用的高质量档案。

### 1.2 工程目标
- **输入容错**：允许“脏输入”进入系统。
- **输出确定性**：给出结构化、可审计、可导出的订单数据。
- **写入幂等**：同一来源文本重复提交，不重复建单。
- **性能可演进**：以当前原型复杂度支持快速迭代，并保留规模化优化空间。

---

## 2. 架构概览（Architecture）

```
用户输入/页面操作
      │
      ▼
FastAPI API Layer (app/main.py)
      │
      ▼
Service Layer (app/service.py)
  ├─ 解析编排（LLM + fallback）
  ├─ 商品简码补全
  ├─ 收件人档案匹配与字段回填
  └─ 业务校验（缺失字段、未匹配商品）
      │
      ▼
Repository Layer (app/repository.py)
      │
      ▼
SQLite (app/db.py)
```

关键设计点：
- 应用级服务缓存，避免请求期重复初始化。
- 数据访问集中在 Repository，隔离 SQL 与业务编排。
- 导出通过模板映射，降低与第三方导入格式耦合成本。

## 3. 关键系统设计决策（Why This Design）

### 3.1 解析策略：LLM + Rule Fallback
- 支持 `llm_only`、`llm_with_fallback`、`fallback` 模式切换。
- LLM 负责语义理解；规则兜底负责在模型不可用时维持可用性。
- 系统返回 `parse_source`，便于线上问题定位和质量评估。

### 3.2 收件人信息补全（解析预览阶段即生效）
- 在解析后即尝试按姓名从收件人库检索候选。
- 通过手机号、身份证、地址关键词、行政区一致性进行打分选择最佳候选。
- 对缺失字段进行“最小侵入式回填”，提升预览可用性和下游建单成功率。

### 3.3 商品简码解析兜底
- 当解析结果缺少 `simple_code` 时，使用商品库做二次匹配。
- 支持别名与口语表达匹配，降低人工修正率。

### 3.4 幂等建单
- 基于 `source_text` 计算 idempotency key（sha256）。
- 重复请求返回已有单号和状态，避免重复写入。

### 3.5 性能演进点（已落地）
- 服务实例缓存，避免每个请求重新初始化依赖。
- 订单写后读取改为精确查询，避免“全量读取后过滤”。
- 增加高频索引（姓名、订单时间+状态、order_items.order_id）。

## 4. 数据模型（Data Model）

核心实体：
- `recipients`：收件人主档（含身份证、省市区、详细地址等）。
- `orders`：订单主表（含来源文本、置信度、状态、幂等键）。
- `order_items`：订单明细（商品简码、数量、阶段等）。
- `product_catalog`：商品词典（商品名 ↔ 简码，含启用状态）。
- `sender_profiles`：寄件人资料（支持默认寄件人）。

建模原则：
- 用唯一约束保证基础数据一致性（如收件人去重键、订单幂等键）。
- 用状态字段而非物理删除表达业务生命周期（部分实体）。

## 5. API 能力边界

### 5.1 当前保留能力
- 文本解析预览
- 从文本建单（含幂等）
- 收件人/订单/商品/寄件人 CRUD
- 收件人图片解析导入
- 模板导出（收件人、订单）

### 5.2 已下线能力
- 寄件人“批量文本导入”和“图片导入”接口及页面入口（按当前业务需求裁剪）。

## 7. 配置策略（Configuration）

关键配置来自 `.env`：
- 解析与模型：`PARSE_MODE`, `LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL`
- 数据库：`DB_PATH`
- 导出模板与目录：`RECIPIENT_TEMPLATE_PATH`, `ORDER_TEMPLATE_PATH`, `EXPORT_DIR`
- 物流默认值：`SENDER_*`, `RECIPIENT_COUNTRY_CODE`, `CHANNEL_CODE`, `GOODS_PURPOSE`

工程建议：
- 本地开发与测试使用独立 `DB_PATH`，避免数据污染。
- CI 默认走 `fallback` 模式，降低外部依赖不确定性。

## 8. 可靠性与可运维性（Ops View）

当前机制：
- 明确的健康检查端点。
- 解析失败区分业务异常（422）与可用性异常（503）。
- 统一的严格类型校验（Pydantic + mypy strict）。

建议下一步：
- 增加结构化日志（请求ID、parse_source、命中策略）。
- 增加关键指标（解析成功率、补全命中率、建单幂等命中率）。
- 引入 lifespan 事件替代 `on_event(startup)` 的弃用写法。

---

## 9. 测试策略（Testing Strategy）

测试重点不是“接口通不通”，而是覆盖系统性风险：
- 解析路径：fallback 地址切分、别名匹配、符号归一化。
- 业务路径：建单幂等、收件人候选打分与复用、缺失字段拦截。
- 管理路径：核心 CRUD 与导出能力。

---

## 10. 系统工程视角下的下一步

如果把这个原型推进到生产级，优先级建议：
1. **可观测性优先**：日志、指标、追踪先行。
2. **边界清晰化**：拆分解析服务与订单服务，降低耦合。
3. **数据演进**：引入迁移工具和版本化 schema。
4. **弹性治理**：为外部 LLM 增加限流、重试预算、熔断策略。
5. **容量治理**：订单列表分页化、热点查询缓存化。
